# usiran-digest-check — Digest 质量检查

当收到 cron 触发消息时激活。本 skill 在每个整点后的第10分钟运行，检查上一个整点（T）生成的 digest 是否符合质量要求。

## 概述

对 `usiran-digest` 定时监控任务的产出做**事后质量审计**，发现不合格 digest 立即修复或删除。

## 本地仓库

- **路径**: `/root/.openclaw/workspace/usiran-digest/`
- **远程**: `git@github.com:fubug/usiran-digest.git`（main 分支）

## 执行流程

### Step 1: 确定检查目标

1. 获取当前北京时间（Asia/Shanghai）
2. 目标 digest 为上一个整点 `T`，即文件 `data/digest/YYYY-MM-DDTHH.md`
   - 例如：当前 10:10 → 检查 T10；当前 03:10 → 检查 T03
3. 用 `read` 工具读取该文件

### Step 2: 判断文件是否存在

- **不存在** → 检查 T-1 是否存在。如果 T-1 也不存在，说明连续两小时无推送，检查 git log 确认是否正常（信源全挂 = 正常跳过）。**输出简短状态日志后结束。**
- **存在** → 进入 Step 3 质量检查

### Step 3: 质量检查（逐项审计）

对 digest 文件执行以下检查，**任何一项不合格即触发修复**：

#### 3.1 时间窗口合规性（红线）

- 提取文件 frontmatter 中的 `id`（如 `2026-04-09T02`），解析为目标时间 T
- **有效窗口**: `[T-1h, T]`，即 01:00-02:00 对应 T02
- **逐一检查**每条事件的时间戳：
  - 如果事件有括号时间标注如 `（02:02）` 或 `(02:02)`，提取该时间
  - 如果事件没有时间标注，但能从上下文推断时间，使用推断时间
  - 如果事件完全没有时间信息，标记为「无时间戳」
- **判断**：
  - 所有有明确时间戳的事件 **必须落在 [T-1h, T] 窗口内**
  - 有时间戳的事件落在窗口外 → **不合格：时间窗口违规**
  - 事件完全没有时间戳，但有真实信源和具体细节 → **轻微不合格：标记但保留**
  - 事件完全没有时间戳且内容空泛（"局势升级""冲突持续"） → **不合格：时间溯源缺失**

#### 3.2 内容实质性检查

- 检查 digest 是否包含**至少1条具体事件**（有明确时间、地点、行为主体）
- **不合格情况**：
  - 全文是"系统维护"、"信息维护时段"、"graceful degradation"等技术性占位内容，无任何实际新闻事件
  - 全文只有"无重大新事件"而没有具体信息
  - 只有信源状态描述（"Web搜索被拦截"）而无新闻内容

#### 3.3 重复性检查

- 读取 T-1 的 digest（如果存在）
- 对比两个 digest 中的事件列表
- **不合格情况**：
  - T 中超过 50% 的事件在 T-1 中已经出现（措辞不同但描述同一事件）

#### 3.4 信源合规性

- 检查 `sources` 字段
- **不合格情况**：
  - 信源只有 `"系统维护"`、`"Generated by usiran-digest automation system"` 等非真实媒体来源
  - 没有任何真实媒体 URL（apnews.com、reuters.com、bbc.com 等）

#### 3.5 格式合规性

- **不合格情况**：
  - 缺少 frontmatter（`---` 包裹的 YAML）
  - `id` 与文件名不一致
  - `date` 不是北京时间格式（应为 `+08:00`）
  - 标题（title.zh / title.en）为空或过短（<5字符）

### Step 4: 修复策略

根据不合格类型执行不同修复：

| 不合格类型 | 修复策略 |
|-----------|---------|
| 时间窗口违规（有时间戳但超出窗口） | 删除违规事件，只保留窗口内事件；如果全部违规则删除整个文件 |
| 无时间戳但有真实信源和实质内容 | **保留**，加审计标记 |
| 无时间戳且内容空泛 | 删除文件 |
| 无实质内容（占位符） | **删除文件**，从 index.json 移除 |
| 重复内容 | 删除重复事件，只保留新增事件；如果无新增则删除文件 |
| 信源不合规 | 删除文件 |
| 格式不合规 | 修正格式 |

#### 4.1 部分修复（有事件可保留）

- 重写 digest 文件，只保留合格事件
- 更新 title 反映实际内容
- 如果只剩1-2条事件，保持简洁，不需要扩充

#### 4.2 完全删除（无合格事件）

```bash
cd /root/.openclaw/workspace/usiran-digest
rm data/digest/YYYY-MM-DDTHH.md
```
然后更新 index.json：

```python
import json
with open('data/digest/index.json') as f:
    idx = json.load(f)
idx['files'] = [e for e in idx['files'] if e['id'] != 'YYYY-MM-DDTHH']
with open('data/digest/index.json', 'w') as f:
    json.dump(idx, f, ensure_ascii=False, indent=2)
```

### Step 5: 完整性验证

修复完成后，执行验证：

1. **文件-索引一致性**：
   ```bash
   # disk 文件数应等于 index.json 中条目数
   ls data/digest/2026-04-*.md | wc -l
   # 用 python 比对文件列表与 index 条目列表
   ```
2. **Git 提交**：
   ```bash
   cd /root/.openclaw/workspace/usiran-digest
   git add -A
   git diff --cached --stat  # 确认变更内容
   git commit -m "check: <简要描述检查结果和修复操作>"
   git push
   ```

### Step 6: 输出审计报告

以简洁格式输出审计结果（供 cron 日志记录）：

```
📋 usiran-digest-check 审计报告 | YYYY-MM-DD TH:10
━━━━━━━━━━━━━━━━━━━━━━━━
目标: T{HH}
结果: ✅ 通过 / ⚠️ 已修复 / ❌ 已删除
检查项:
  时间窗口: ✅/❌ (说明)
  内容实质: ✅/❌
  重复性:   ✅/❌
  信源合规: ✅/❌
  格式合规: ✅/❌
修复操作: (如有) <描述>
```

## 特殊情况

### 连续无推送

如果 T 和 T-1 都不存在：
- 检查 git log 最近3次提交，确认是正常跳过（信源不可用）
- 不触发告警，这是预期行为
- 输出：`✅ 连续跳过（信源不可用），无需干预`

### Git 无变更

如果检查通过或文件不存在且无需修复：
- 不执行 git 操作
- 输出：`✅ T{HH} 审计通过，无需修复`
